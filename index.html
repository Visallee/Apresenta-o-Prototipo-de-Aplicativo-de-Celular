▸ AMADURECIMENTO
Evolução da Arquitetura

Com a base validada, o projeto evoluiu naturalmente: adotamos padrões de mercado para garantir escalabilidade e funcionamento offline robusto.

  • Separação de responsabilidades com MVVM
  • Banco de dados estruturado via Drift / SQLite — 100% offline
  • Estado global reativo com Riverpod — sem setState excessivos
ANTES Arrays voláteis, setState em cascata, lógica de negócio misturada na View
↓
DEPOIS SQLite persistente, Riverpod reativo, camadas independentes e testáveis
Padrão Arquitetural
Arquitetura MVVM na Prática

A View não sabe como o banco funciona — ela apenas reage às variáveis entregues pela ViewModel. Dividimos o app em 3 camadas independentes.

CORE / DATA
Acesso aos dados
  • Tabelas Drift
  • AppDatabase
  • JogoRepository
→ watch/read ← invalidate
VIEWMODEL
Lógica de negócio
  • Riverpod Providers
  • JogoViewModel
  • FutureProvider
→ rebuild ← eventos
PRESENTATION
Exibição e interação
  • ConsumerWidget
  • HomePage
  • GameCard
Camada Core
Banco de Dados — Drift

O Drift gera nossa tabela Jogos no SQLite. O modelo é tipado e seguro — sem erros de coluna em tempo de execução.

IntColumn → INT com autoIncrement (PK)
TextColumn → VARCHAR com validação de tamanho
RealColumn → REAL para a nota decimal
@DataClassName → Nomeia a classe gerada
Dart · Drift
@DataClassName('Jogo')
class Jogos extends Table {
  IntColumn get id
    => integer().autoIncrement()();

  TextColumn get nome
    => text().withLength(
         min: 1, max: 120)();

  TextColumn  get status
    => text()();

  IntColumn  get conquistasAtuais
    => integer()();

  IntColumn  get conquistasTotal
    => integer()();

  RealColumn get nota
    => real()();
}
Camada Data
O Repositório

O JogoRepository encapsula todas as consultas ao banco — a ViewModel nunca toca o SQLite diretamente.

  • Ordenação por nome garantida no repositório
  • Inserção via JogosCompanion — tipado e seguro
  • Fácil de trocar a fonte de dados no futuro
Dart · Repository
Future<List<Jogo>> listarJogos() {
  return (database
    .select(database.jogos)
    ..orderBy([(t) =>
      OrderingTerm.asc(t.nome)])
  ).get();
}

Future<void> cadastrarJogo(
    JogosCompanion jogo) async {
  await database
    .into(database.jogos)
    .insert(jogo);
}
Camada de Negócio
Riverpod & ViewModel

A JogoViewModel conecta repositório e interface. Após salvar, ref.invalidate avisa toda a árvore de widgets para se reconstruir.

FLUXO REATIVO View aciona ação → ViewModel salva no Repositório → ref.invalidate() → Provider recarrega → View atualiza sozinha
Dart · Riverpod
final jogosProvider =
  FutureProvider<List<Jogo>>(
    (ref) async {
      final repo = ref.watch(
        jogoRepositoryProvider);
      return repo.listarJogos();
  });

// Na ViewModel, após inserir no banco:
ref.invalidate(
  jogosProvider); // ← Notifica a View!
Camada Presentation
Interface Reativa

A HomePage usa ConsumerStatefulWidget e reage automaticamente aos 3 estados do Provider.

loading → Exibe CircularProgressIndicator
error → Exibe mensagem de erro descritiva
data → Filtra por aba e exibe lista de jogos
Dart · Flutter
final jogosAsync =
  ref.watch(jogosProvider);

body: jogosAsync.when(

  loading: () => const
    CircularProgressIndicator(),

  error: (e, stack) =>
    Text('Erro: $e'),

  data: (jogos) {
    // Filtra e exibe na TabBarView
  },
)
